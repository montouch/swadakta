# Swadakta First Paid Pilot Script

Use this script before public paid launch. It is designed for one low-value internal or friendly-client job that proves Swadakta works end to end without creating avoidable legal, payment, identity, or safety risk.

## Pilot Job Choice

Choose a simple, legal, low-risk, easy-to-prove job:

- local site photo check
- supplier price check
- simple delivery quote
- document pickup quote
- family-support errand with clear consent
- virtual-assistant research task

Do not use the first pilot for land/title transfer, cash collection, high-value goods, customs-uncertain items, medical/legal advice, restricted goods, authority-sensitive documents, or anything that feels unclear.

## Hard Rule

The pilot can prove the workflow. It does not authorize Swadakta to act as a licensed escrow provider, remittance provider, bank, law firm, customs broker, immigration adviser, or guaranteed delivery service.

AI may draft, organize, summarize, and prompt. AI must not approve ID, mark funds paid, assign receivers, release/refund money, or make legal/customs/payment commitments.

## Step-by-Step Rehearsal

1. Create or sign in as the client at `https://swadakta.com/portal`.
2. Save account profile basics: name, email, mobile backup, base country, preferred currency, and role choice.
3. Refresh the page and confirm the account returns to Account Home instead of staying on sign-in.
4. Create the pilot request at `https://swadakta.com/brief`.
5. Include origin, destination, task location, budget comfort, proof requirements, contact preference, and compliance acknowledgement.
6. Open admin at `https://swadakta.com/admin-ops` and confirm the request code appears.
7. Run safe autopilot only for drafting/classification. Keep protected decisions manual/provider-gated.
8. Start ID verification from Account Home or `https://swadakta.com/verification`.
9. Confirm the request either receives a provider reference/link or stays clearly queued with paid actions locked.
10. Quote the job in admin: amount, currency, service fee wording, payment preference, payment due date, protected amount, funds status, release condition, and at least two milestones.
11. Create payment via the configured low-risk rail: Stripe, PayPal, M-Pesa sandbox/live, or fallback payment link only if approved for the pilot.
12. Confirm payment evidence: provider reference, amount, currency, payer, and request code.
13. Do not assign a receiver or release money from payment evidence alone.
14. Have a receiver profile apply or place an offer from `https://swadakta.com/portal#find-work`.
15. Confirm unverified receivers cannot be assigned.
16. Choose the receiver by trust, fit, proof plan, route suitability, and provenance, not just lowest price.
17. Send one test message/update and one proof upload/link from `https://swadakta.com/messages`.
18. Review proof in admin before making it client-facing.
19. Confirm tracking at `https://swadakta.com/tracking` shows safe request status, funds status, milestones, proof/report link, and resolution path without exposing internal notes or founder economics.
20. Close the job only after proof is accepted, review is captured, and there is no unresolved dispute.

## Pass Conditions

- Signed-in account reaches Account Home and stays signed in after refresh.
- Brief creates a request code and admin can see route, proof, budget, and consent context.
- Identity handoff creates provider evidence or a clear queued state.
- Paid or sensitive actions stay locked while ID evidence is missing.
- Payment evidence matches provider reference, amount, currency, payer, and request code.
- Receiver assignment is blocked unless the receiver is vetted and ID-verified.
- Proof and messages are admin-reviewable before client-facing closeout.
- Tracking shows safe status without exposing internal notes or founder economics.
- AI helps but does not approve ID, release/refund money, assign receivers, or make protected external commitments.

## Fail and Pause Conditions

Pause the pilot if any of these happen:

- sign-in loops back to the sign-in page
- profile save fails
- request code is missing
- ID provider route is missing for a paid/sensitive job
- amount or currency does not match provider evidence
- provider reference cannot be reconciled
- receiver can be assigned while unverified
- proof is missing, unclear, mismatched, or exposes unnecessary personal data
- client cannot track the job
- any dispute, legal uncertainty, customs uncertainty, restricted-goods concern, or safety concern appears

When a fail condition appears, mark the job as paused/disputed/manual-review in admin and do not accept similar paid jobs until the issue is fixed and retested.
